# Semantic Analysis Prompt

You are analyzing a Swift iOS codebase to identify repository-level issues that require cross-file understanding.

## Task

Analyze the codebase using the data from Phases 1-3 and identify:
- **Bugs**: Logic errors, potential crashes, race conditions, memory issues
- **Code smells**: Violations of SOLID principles, tight coupling, poor separation of concerns
- **Architecture issues**: Inconsistent patterns, mixed paradigms (UIKit/SwiftUI), circular dependencies
- **Test gaps**: Untested critical code, missing edge case tests, no integration tests
- **Documentation gaps**: Complex code without comments, public APIs without docs, unclear responsibilities

## Input Data

You have three sources of information:

### 1. File Summaries (Phase 3 - AI-generated summaries)
```json
{file_summaries}
```

These summaries describe each file's role, types, dependencies, and responsibilities.

### 2. Static Findings (Phase 2 - Syntactic analysis)
```json
{static_findings}
```

These are crash risks and structural smells detected by static analysis (force unwraps, singletons, large classes).

### 3. Dependency Graph (Phase 2 - Import/type relationships)
```json
{dependency_graph}
```

This maps each file to its dependencies (imports and referenced types).

## Analysis Guidelines

1. **Repository-level thinking** - Look for patterns across multiple files. Static analysis already caught single-file issues.

2. **Evidence-based findings** - Every finding must cite specific files, line numbers (from static findings), or concrete evidence from the summaries.

3. **Architectural perspective** - Identify:
   - Circular dependencies between files
   - Mixed architectural patterns (e.g., UIKit view controllers calling SwiftUI views)
   - Violation of separation of concerns (e.g., UI code in data models)
   - Missing abstraction layers (e.g., views directly accessing persistence)

4. **Testing perspective** - Identify:
   - Critical business logic without tests
   - Error handling code paths that are hard to test
   - Singletons that make testing difficult
   - Missing integration tests for multi-file workflows

5. **Severity classification**:
   - **high**: Crashes, data loss, security issues, critical bugs
   - **medium**: Code smells affecting maintainability, test gaps for important features
   - **low**: Minor architectural inconsistencies, documentation gaps

6. **Actionable recommendations** - Provide specific, implementable fixes (e.g., "Extract NetworkService protocol to break dependency cycle between Store.swift and API.swift").

## Expected Output Format

Return a valid JSON array of findings matching this schema:

```json
[
  {{
    "type": "string",
    "location": "string",
    "explanation": "string",
    "confidence": "high|medium|low",
    "subtype": "string",
    "severity": "high|medium|low",
    "evidence": "string",
    "recommendation": "string"
  }}
]
```

### Field Descriptions

**REQUIRED FIELDS (per assignment spec):**
- **type**: Category of issue - one of: "bug", "smell", "architecture", "test_gap", "doc_gap", "uncertainty"
- **location**: Where the issue occurs - format as "file:line" (e.g., "Store.swift:24") or "file:line1,line2,line3" for multiple lines, or "ComponentName" for architectural issues
- **explanation**: Clear, concise description of the issue (1-2 sentences)
- **confidence**: How certain you are this is a real issue - "high", "medium", or "low"

**OPTIONAL BONUS FIELDS (provide extra value):**
- **subtype**: Specific issue type (e.g., "force_unwrap", "circular_dependency", "singleton_testability")
- **severity**: Impact level if true - "high" (crashes, data loss), "medium" (maintainability), "low" (minor)
- **evidence**: Specific evidence from the data - cite file names, line numbers from static findings, dependencies
- **recommendation**: Actionable fix with concrete steps
